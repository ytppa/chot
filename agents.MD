# Nothing Chat - Agent Guidelines

Этот файл фиксирует правила для Codex/агентов, работающих над проектом Nothing Chat.
README содержит подробный контекст, а здесь собраны принципиальные решения, которых нужно придерживаться при разработке.

## Рабочая папка

- Работать только в `C:\Games\MySandbox\nothing-chat`.
- Не писать в старый путь `C:\Users\AGA\Documents\Ничтофон`.
- Не переносить изменения из старой копии без явного запроса пользователя.
- Перед Git-операциями проверять текущее состояние рабочей директории.

## Цель MVP

- Делать веб-чат на WebSocket, двигаясь от простого к сложному.
- Первый MVP: регистрация, логин, ручное подтверждение пользователей админом, личные чаты, plain text сообщения.
- Группы, каналы, вложения, картинки, ответы, цитаты, пересылки, markdown и форматирование текста отложены.
- Почту пока не подключать; позже она понадобится для уведомлений и восстановления пароля.

## Принципиальный стек

Frontend:

- TypeScript.
- Vite.
- Нативные Web Components.
- Shadow DOM использовать выборочно, там где нужна инкапсуляция.
- Не использовать React как основной UI-слой.
- Lit можно добавить позже точечно, если ручной код компонентов станет слишком шумным.

Backend:

- Node.js LTS.
- Fastify.
- `@fastify/websocket` поверх `ws`.
- PostgreSQL.
- Drizzle ORM.
- Zod или Valibot для валидации HTTP payload и WebSocket протокола.
- `argon2id` для паролей.
- Cookie-based sessions: opaque token в `HttpOnly` cookie.

## Структура проекта

Базовое направление структуры:

```text
apps/
  web/
    src/
      app/
      services/
      components/
      utils/
      styles/

  server/
    src/
      db/
      modules/
      ws/
      http/

packages/
  shared/
    src/
```

- `apps/web` отвечает только за браузерный UI.
- `apps/server` отвечает за HTTP API, WebSocket, auth, работу с БД и бизнес-логику.
- `packages/shared` хранит общие типы, DTO, схемы валидации и WebSocket protocol types.
- Не размазывать общие типы между frontend и backend вручную; выносить их в `packages/shared`.

## Backend подход

- HTTP использовать для auth, initial load, admin actions и пагинации истории.
- WebSocket использовать для realtime-событий.
- На каждом HTTP и WS действии проверять, что пользователь имеет доступ к нужному ресурсу.
- Не доверять client payload; все входящие данные валидировать.
- Для повторной отправки сообщений использовать `clientNonce`, чтобы не создавать дубли.
- Сессии хранить через hash токена, сам session token в БД не сохранять.
- Админ создается seed-логикой из env: `ADMIN_LOGIN`, `ADMIN_PASSWORD`.

## Frontend подход

- UI строить на custom elements: `x-app-shell`, `x-login-form`, `x-chat-list`, `x-message-list`, `x-message-bubble`, `x-message-composer`.
- Пользовательский текст никогда не рендерить через `innerHTML`.
- Для текста сообщений использовать `TextNode`.
- Ссылки парсить отдельно и рендерить через `<a target="_blank" rel="noopener noreferrer">`.
- События из Shadow DOM, которые должны всплывать наружу, отправлять с `bubbles: true` и `composed: true`.
- Store на старте может быть простым observable/EventTarget store.
- Не раскладывать overlay, popup и context menu логику по отдельным компонентам; держать единый верхнеуровневый слой.

## Контекстное меню

- В приложении должен быть один `x-context-menu-root` на верхнем уровне.
- Компоненты открывают меню через событие `app-context-menu`.
- Команды собираются через registry по типу сущности, правам пользователя и текущему состоянию.
- Меню должно закрываться по Escape, outside click, scroll и resize.
- Для touch-сценариев предусмотреть long press.

## Данные и сообщения

- Сообщения хранятся как plain text.
- HTML в БД не хранить.
- `entities` можно хранить как JSONB с диапазонами ссылок.
- Вложения и отношения между сообщениями не смешивать:
  - файлы идут в отдельную сущность `message_files`;
  - reply/forward/quote идут в отдельную сущность `message_relations`.

## Безопасность

- Пароли только через `argon2id`.
- Session token в cookie должен быть `HttpOnly`.
- Для HTTPS включать `Secure`.
- Использовать `SameSite=Lax`.
- Ограничивать размер WebSocket payload.
- Добавлять rate limit для login, register и message sending.
- Пользовательский контент не вставлять через `innerHTML`.
- Ошибки API и WS не должны раскрывать чувствительные детали.

## Стиль кода

- Писать TypeScript строго и явно: не использовать `any`, если можно выразить тип нормально.
- Держать модули маленькими и связанными с одной зоной ответственности.
- Предпочитать простые функции и понятные типы ранним абстракциям.
- Не добавлять новый фреймворк или крупную библиотеку без веской причины.
- Повторяющиеся контракты, типы событий и DTO выносить в `packages/shared`.
- Имена файлов и компонентов должны отражать назначение, а не техническую случайность.

## Комментарии в коде

- Каждый метод или функция должны сопровождаться емким комментарием, объясняющим назначение метода на уровне смысла.
- Комментарий должен отвечать на вопрос "зачем это нужно", а не пересказывать каждую строку.
- Сложные методы разделять на логические блоки.
- Перед каждым нетривиальным логическим блоком добавлять краткий комментарий, который объясняет роль блока.
- Не писать шумные комментарии вроде "увеличиваем счетчик" или "возвращаем результат", если это очевидно из кода.
- Если код стал требовать слишком много пояснений, сначала подумать об упрощении структуры.

Пример желаемого стиля:

```ts
/**
 * Restores the current user from a session cookie and rejects inactive accounts.
 */
async function resolveCurrentUser(request: FastifyRequest): Promise<AuthUser | null> {
  // Extract and verify the opaque session token before touching user data.
  const sessionToken = readSessionCookie(request);
  if (!sessionToken) {
    return null;
  }

  // Load the session owner and ensure the account is allowed to use the app.
  const user = await findUserBySessionToken(sessionToken);
  if (!user || user.status !== 'active') {
    return null;
  }

  return user;
}
```

## Миграции и база

- Схему БД описывать через Drizzle.
- Миграции должны быть проверяемыми и воспроизводимыми.
- Индексы и уникальные ограничения добавлять вместе с сущностями, для которых они принципиальны.
- Не хранить вычисляемое UI-состояние в БД без необходимости.

## WebSocket протокол

- Все события использовать через общий envelope.
- Client events должны иметь `id`, `type`, `payload`.
- Server events должны иметь `type`, `payload`, `ts`.
- Ошибки возвращать структурированно, без произвольных строк как единственного контракта.
- Изменения протокола сначала отражать в `packages/shared`, потом использовать в server и web.

## UX и UI

- Первый экран должен быть рабочим интерфейсом, а не лендингом.
- Интерфейс чата должен быть спокойным, плотным и удобным для повторного использования.
- Не перегружать MVP декоративными элементами.
- Для длинных списков сообщений заранее держать в голове будущую виртуализацию.
- Контент не должен прыгать при hover, loading states или изменении счетчиков.

## Документы проекта

- `README.md` хранит большой контекст, причины решений и справочную информацию.
- `agents.MD` хранит принципиальные правила для агентов: стек, архитектурные границы, безопасность и стиль работы.
- `PLAN.md` хранит актуальный план разработки, статусы этапов и ближайший фокус.
- Если в ходе работы появляется новый принципиальный момент, который должен влиять на будущие решения, добавлять его в `agents.MD`.
- Если начинается, завершается, блокируется или откладывается значимый этап, обновлять статус в `PLAN.md` в той же задаче.
- Не дублировать все подряд между документами: README объясняет контекст, `agents.MD` задает правила, `PLAN.md` отслеживает движение.
- Перед началом крупной задачи сверяться с `PLAN.md`, а перед завершением проверять, нужно ли скорректировать статусы.
- Не оставлять план устаревшим после изменений в структуре, стеке или очередности разработки.

## Сомнения по реализации

- Если у агента есть сомнения по реализации, архитектуре, UX, модели данных, безопасности или выбору библиотеки, всегда спрашивать пользователя перед изменением кода.
- Не принимать молча спорные решения, которые могут повлиять на дальнейшее развитие проекта.
- Если вопрос мелкий и не влияет на поведение, архитектуру или пользовательский опыт, можно выбрать самый простой вариант и кратко зафиксировать это в ответе.
